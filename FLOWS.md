### WhatsApp Results flow
`рЯТђ WhatsApp Results` button (admin, Exams page) вЖТ `WhatsAppPreviewModal` (review + edit) вЖТ `POST /api/send-whatsapp` вЖТ `WhatsAppResultsModal` (log).

**Pre-send modal** (`WhatsAppPreviewModal.jsx`): rows built from `exam.students` + `studentProfiles`; branch dropdown (derived from `studentProfiles`), mobile + parent mobiles editable inline. Footer has optional "redirect all to" test field. On "Confirm Send": calls `bulkUpdateStudentContacts(edits)` (single fetchвЖТpatchвЖТwrite to `students_db.json`), then POSTs `{ examName, redirectTo?, students? }` to `/api/send-whatsapp`.

**Send history** (`whatsappSendHistory` store key, `{ [examId]: { sentAt, sent, skipped, failedNames[] } }`): persisted to `faculty-data.json`. Button shows `рЯТђ Sent NвЬУ MвЬЧ ¬Ј Resend` after first send. `failedNames[]` is parsed from log lines client-side (`SKIP Name вАФ` and `FAIL вЖТ Name (student`).

**Resend scope toggle**: when `failedNames` is non-null (previous send exists), modal shows amber banner with radio: "Failed & skipped only (N)" (default) / "All students". Scope controls the `students[]` array forwarded to `--students` in the script.

**`--students` filter** in `send_results_whatsapp.py`: comma-separated names, case-insensitive; filters `results` list before the send loop. Forwarded from POST body by the Vite dev endpoint or the `api/send-whatsapp.js` Vercel serverless function вАФ same URL (`/api/send-whatsapp`) in both environments.

### Exam absences вАФ persistent event log

`exam_absences` is a sparse event log: one row per (exam_id, lws_id) for every student in the expected cohort who didn't appear in `exam.students[]`. Columns: `id`, `exam_id` (FK ON DELETE CASCADE), `lws_id` (FK ON DELETE CASCADE), `marked_at`, `marked_by` (`'upload'` by default), `notified_at` (set when the WhatsApp absence alert lands). UNIQUE `(exam_id, lws_id)`. Indexed on `exam_id`, `lws_id`, and `(lws_id, marked_at DESC)`. RLS: authenticated only.

**Auto-sync on upload.** `examsSlice.addExam` and `replaceExam` both call `get().syncExamAbsences(examId)` after the local state write. The sync helper computes the current absentee set via `getExamAbsentees(exam, studentProfiles)`, reads existing `exam_absences` rows for the exam, and reconciles asymmetrically: **DELETE only for students who now appear in `exam.students[]`** (i.e. re-upload reveals they attended after all); INSERT for new absentees; **leave alone** any existing row whose student fell out of the cohort but didn't attend (batch moves, profile deletions) вАФ those historical absences are still factual. Preserves `notified_at` across re-uploads. Returns `{ added, removed, kept }`.

**Self-heal on first modal open.** `ExamAbsencePreviewModal` reads from `getExamAbsencesForExam(examId)` on mount. If the table is empty (legacy exam uploaded before this feature), it triggers `syncExamAbsences(examId)` once and re-fetches. Replaces the previous inline computation and means legacy exams get backfilled lazily as faculty opens them вАФ no migration script needed.

**Surfacing per-student.** `MissedExams.jsx` renders a card on `StudentView` listing absences (enriched with exam name, date, batch) sorted newest-first; hidden when zero. `RecentIncidents` extends to include exam absences in its last-30-days strip as `Missed exam ¬Ј {examName}` chips. Both components support a `examAbsencesProp` prop bypass for the student portal (no Supabase session, so the slice fetch would return [] вАФ `api/student-login.js` returns `examAbsences[]` (last 30 days) which threads through App.jsx вЖТ StudentView вЖТ MissedExams/RecentIncidents).

**Notified audit.** After a successful absence-alert send, `Exams.jsx`'s `handleExamAbsenceConfirm` computes `notifiedLwsIds = edits.filter(e => !failedSet.has(e.name)).map(e => e.lwsId)` and calls `markExamAbsencesNotified(examId, notifiedLwsIds)` to stamp `notified_at`. Rows whose student leg OR any parent leg failed are NOT marked (they appear in `failedNames` via `parseFailedNamesAbsence`). The modal shows a green "Notified" badge per row whose `notified_at` is set.

### Exam absence alert flow
`рЯУµ Send Absent Alert` button (admin, Exams page, next to `рЯТђ WhatsApp Results`) вЖТ `ExamAbsencePreviewModal` вЖТ `POST /api/send-exam-absence` вЖТ `WhatsAppResultsModal`.

**Multi-batch exams.** An exam may be sat by вЙ•1 batch. Step 2 of the upload modal stores the central batches as a **comma-joined string** in `exam.batch` вАФ e.g. `"APJ_NDA_2Y_(26-28), LWS_NDA_2Y_(26-28)_A"` for an exam sat by two cohorts together. Readers split via the pure `getExamBatches(exam)` helper in `src/lib/analytics/filters.js`. Single-batch exams are just the bare name with no comma; legacy exams with HR-namespace `exam.batch` parse to a single (stale) element.

**Cohort derivation.** `getExamAbsentees(exam, studentProfiles)` returns profiles whose `batches[] вИ© getExamBatches(exam) вЙ† вИЕ` AND whose canonical name (or any `nameVariants[]` entry) is NOT in `exam.students[]` AND whose `accountStatus === 'Active'` AND whose `regDate` (if set) is on/before `exam.date`. Variant-keyed entries in the profiles map are skipped via `p.name === key`, so each absentee is returned exactly once. Empty when `exam.batches` is empty (legacy exam) or no profile matches. **EIS+Active gate**: demo students (never registered вЖТ not in `studentProfiles`) are excluded structurally; Block / quit / batch-over students are excluded by the `accountStatus` check. **regDate gate**: students registered after the exam date are excluded (they weren't enrolled yet); missing `regDate` is permissive (no filter applied вАФ matches `filterValidExams`).

**Modal-layer EIS+Active filter.** `ExamAbsencePreviewModal`'s `joinRows` independently filters historical `exam_absences` rows: drops rows whose joined profile is missing OR whose `accountStatus !== 'Active'`. Needed because a student may have been Active when the absence was recorded but Block now вАФ the audit log preserves the historical fact, but the WhatsApp send must only reach currently-Active students.

**Student + parents.** Endpoint sends to **both** the student's own mobile and every entry in `parentMobiles[]` вАФ accountability signal so the student knows their parents are being informed (decisions log). Template body still reads "Your ward was absentвА¶" (Meta-approved, neutral re-wording deferred); slight off-tone on the student-copy was accepted in exchange for shipping today.

**Template (Wabridge / Meta).** Body uses positional `{{1}} = name`, `{{2}} = examName`. `examName` is ASCII-sanitised in `api/send-exam-absence.js` вАФ en-dash/em-dash вЖТ `-`, newlines вЖТ space, runs of whitespace collapsed. Same Meta silent-drop rules as the other flows; see `feedback_whatsapp_template_param_rules`. Env: `WABRIDGE_EXAM_ABSENCE_TEMPLATE_ID`.

**Send history**: `examAbsenceSendHistory[examId] = { sentAt, sent, skipped, failedNames[] }`, persisted via `saveToStorage` (added to both the destructure and the data object). Button shows `рЯУµ Sent NвЬУ MвЬЧ ¬Ј Resend` after first send. `failedNames[]` parsed via `parseFailedNamesAbsence` in `Exams.jsx` вАФ captures `FAIL вЖТ Name (parent`, `SKIP Name parent ...`, and `SKIP Name вАФ` lines (separate parser from the WhatsApp Results flow which only matches `(student`).

**Resend scope toggle**: when `failedNames` non-null, modal shows the amber `Failed & skipped only (N) / All students (M)` radio, defaulted to failed-only (same shape as `LateNotificationPreviewModal`). `bulkUpdateStudentContacts(edits)` runs before send to persist any parent-mobile edits.

### Exam integrity вАФ copying detection + admitted incidents (2026-06-20)

**Detection (read-only).** The рЯХµ Integrity toggle on each exam card (admin + teacher) renders `buildExamIntegrityReport(exam)` from [`src/lib/analytics/examIntegrity.js`](src/lib/analytics/examIntegrity.js) вАФ a pure pairwise analysis over the captured chosen options (`exam_results.choices`, available only on uploads since 2026-06-10). The fingerprint of copying is **shared WRONG answers** (same incorrect option on the same question) + near-identical attempt/skip patterns, NOT shared correct answers. Two regimes: **Tier A** near-identical papers (`diff вЙ§ 5 && sameWrong вЙ• 8`) catches clusters; **Tier B** z-score outliers (`z вЙ• 4`) **gated by Harpp-Hogan ratio вЙ• 1** so a weak-but-honest student who hits popular distractors with many peers (a high-`diff` "hub") is not flagged. Union-find groups flagged pairs into clusters/rings; roll-adjacency is a corroborating pill. Per-pair **Evidence** drill-down reuses `QuestionCard` to show exactly which questions both got identically wrong. `available:false` (with a notice) for offline exams and pre-2026-06-10 uploads. Output is **investigative leads, not proof** вАФ stated in the panel.

**Capture вЖТ incident.** A teacher confronts the flagged student(s) offline. If a student **admits**, the teacher clicks the one-click **"[name] admitted"** action on that pair row (a `window.confirm` guards the misclick, because only an admin can later delete). The panel resolves the exam-sheet name вЖТ `lwsId` via `studentProfiles` (canonical + variants, lowercased) and calls `integritySlice.logIntegrityIncident({...})`, which upserts an `integrity_incidents` row (one per student/exam; counterpart = the other flagged student). **Evidence (shared-wrong / diff / both-answered) and exam name/date are snapshotted onto the row** so the record stays factual even if the exam is re-uploaded or deleted (that's why `exam_id` has no FK). A "вЬУ logged" badge then shows on that student, fed by `getIntegrityIncidentsForExam(examId)`.

**Surfacing.** `IntegrityIncidents.jsx` renders a red "вЪ† Academic Integrity" card on `StudentView` (hide-when-empty, like `MissedExams`) listing exam ¬Ј date ¬Ј counterpart ¬Ј evidence ¬Ј note ¬Ј recorder. Admin/teacher fetch via `getIntegrityIncidentsForStudent(lwsId)`; **the student/parent portal sees it too** via `integrityIncidentsProp` вАФ `api/student-login.js` returns `integrityIncidents[]` (self-contained rows, no join) which threads App.jsx вЖТ StudentView вЖТ the card. **Delete (√Ч) is admin-only** (`mode === 'admin'`); a teacher can log but not void a disciplinary record.

**Non-goal.** Logging an incident does NOT change the student's marks вАФ Evalbee-authoritative grading is untouched (a score consequence would be a separate, deliberate decision).

### Late marking & lecture-miss flow (Attendance page)

Two related signals beyond daily P/A, both surfacing on the Attendance page (`src/pages/Attendance/index.jsx`):

**(1) Late to first lecture** вАФ sparse, in-app marking. `student_attendance.status='L'` replaces the daily P/A for that (lws_id, date). Faculty marks via the top-of-page `LateMarkingWidget` (admin only): search вЖТ add вЖТ chip list вЖТ "Send Morning Late Notifications" button. Actions `markLate(lwsId, date)` / `unmarkLate(lwsId, date)` / `getLateStudentsForDate(date)` live on `attendanceSlice.js`. `importAttendance` queries existing L rows for the (lws_id, date) pairs in the XLS and filters them out before upsert вАФ without this, the morning marking would be silently overwritten when the LWS attendance XLS imports at end of day (the XLS only carries P/A/-). Returns `lateProtected` count so the success banner can surface preserved markings.

**(2) Per-lecture absences during the day** вАФ sparse event log in `lecture_absences`. Faculty enters via the **Lecture log tab** on the Attendance page: pick date + batch вЖТ per-period cards (derived from `getTodaysLectures(timetable, date, mappings)`) вЖТ click "Mark absentees" вЖТ modal with searchable multi-select вЖТ save. Replace-set semantics per period card via `lectureAbsenceSlice.setLectureAbsenteesForPeriod(date, slotId, subject, lwsIds)` which deletes existing rows for `(date, slot_id)` then inserts the new set. Keying by **slot_id** (not subject) means two same-day same-subject periods (e.g. Maths at 11 AM + Maths at 2 PM) stay independent вАФ see decisions log. Subjects come from the batch's timetable for that weekday, NOT from a hardcoded list. **Impromptu lectures** (2026-06-06): a "+ Add impromptu lecture" form (subject required, start/end time optional) covers lectures not in the timetable вАФ it mints an `adhoc_*` `slot_id` and optionally persists the time via the new `start_time`/`end_time` columns; the card then behaves like a timetabled one (mark absentees, send) plus a remove √Ч, and is reconstructed on reload from its `adhoc_*` rows (subject + time live on the row, so no timetable is needed). The "+ Add" affordance shows even when nothing is timetabled вАФ so a batch with no timetable, or **Sunday/holidays** (where `getTodaysLectures` returns `[]`), can still log an impromptu-lecture miss.

**Pure helper** `src/lib/timetable.js` вЖТ `getTodaysLectures(timetable, date, mappings)` returns `[{ slotId, startTime, endTime, subject, mappingId, label }]` in time order. Skips `__span` rows (lunch), breaks, slots with no entry for the resolved day, and class cells whose `mappingId` no longer exists in `mappings`. Accepts either a `Date` object or a `YYYY-MM-DD` ISO string; ISO strings are parsed as local dates so `getDay()` returns the local weekday.

**Sending** вАФ two Vercel serverless endpoints mirror the `api/send-whatsapp.js` pattern: `api/send-late-notifications.js` (variables `[name, date]`) and `api/send-lecture-absences.js` (variables `[name, date, formattedSubjects]`). Both verify the admin JWT (Bearer header), then loop `students[]` from the request body, sending one Wabridge call per student mobile + each parent mobile. `redirectTo` overrides every destination (test-mode). Template IDs come from `WABRIDGE_LATE_TEMPLATE_ID` and `WABRIDGE_LECTURE_MISS_TEMPLATE_ID` env vars; endpoints return `500` with a helpful message if missing (Meta template approval is parallel вАФ UI flows still work end-to-end without templates so dev can rehearse the marking + preview without firing real messages). The Vite dev plugin (`vite.config.js`) has a small shim that imports the same JS handler and adapts the req/res shape, so the endpoints work in both dev and prod under the same URL.

**Wabridge / Meta template parameter rules** (learned the hard way; see decisions log + memory `feedback_whatsapp_template_param_rules`):
- Template body must use **positional** `{{1}} {{2}} {{N}}` placeholders вАФ Meta does NOT substitute named placeholders like `{{name}}`.
- Variable values must be **ASCII-only**: no en-dash `вАУ`, em-dash `вАФ`, or other Unicode punctuation (Meta drops messages silently).
- No newlines, tabs, or 5+ consecutive spaces inside a variable value (same вАФ silent drop).
- Avoid parentheses-around-colons patterns like `(11:00 AM - 12:30 PM)` вАФ Meta's "looks-like-rich-formatting" filter rejects them. `api/send-lecture-absences.js` formats the per-subject string as `Subject HH:MM AM to HH:MM PM` (plain ASCII, comma-joined for multiple subjects).

**Pending-aware resend** (late + lecture-miss + homework, 2026-06-05) вАФ each send records *who/what was actually notified*, so students marked late (or logged) AFTER a send are never silently skipped. After a send, AttendancePage parses `data.lines` via `parseFailedNames()` (`FAIL вЖТ Name (student|parent)` / `SKIP Name вАФ|parent`, exported from `src/pages/Attendance/index.jsx`), computes the **successfully-notified set** (`payload.students вИТ failedNames`), and merges it into the per-day/per-batch history entry:
- **Late**: `lateSendHistory[YYYY-MM-DD] = { sentAt, sent, skipped, failedNames, notifiedLwsIds }`.
- **Lecture-miss**: `lectureMissSendHistory[\`${date}|${batchName}\`]` вАФ same shape (compound key keeps two batches independent).
- **Homework**: `homeworkSendHistory[\`${date}|${batchName}\`]` stores **`notifiedItemKeys`** (`homeworkNotifyKey(lwsId, subject, chapter, type)` = `lwsId|||subject|||chapter|||type`) вАФ **item-level**, because homework sends one message per (student, item); a new item on an already-notified student must still surface. Server-side `notified_at` is still stamped via `markHomeworkNotified` as an independent audit.

`pending` = currently-marked students/items whose id/key is NOT in the notified set вАФ it **subsumes both never-sent (added after the send) AND previously-failed legs** (the old "Failed-only (0)" default was a trap: a student added after the send got nothing unless faculty manually flipped scope to All). The widgets (`LateMarkingWidget`, `LectureLogTab`, `HomeworkLogTab`) render three button states off `pending`: first send (`Send вА¶`), **`Notify N pending`** (primary, when pending > 0), or **`вЬУ All N notified ¬Ј Resend all`** (secondary). Late chips show a per-chip вЬУ for notified students. The preview modals take `notifiedLwsIds` (late/lecture) or `notifiedItemKeys` (homework вАФ filtered per row, so a partially-notified student sends only their un-notified items) and default the scope toggle to **Pending only (K)** / All (N) вАФ so a late-after-send student goes out with one click and the already-notified get no duplicate. All three modals call `bulkUpdateStudentContacts(edits)` before send.

**Preview modals** (`LateNotificationPreviewModal`, `LectureMissPreviewModal`) mirror the existing `WhatsAppPreviewModal` shape: editable mobile + parent_mobiles per row, redirect-to test field at the bottom, Confirm / Cancel. The lecture-miss modal additionally shows the missed-subjects list per student. Empty input вЖТ Confirm disabled.

**Recent incidents** вАФ `src/pages/Students/RecentIncidents.jsx` sits below `AttendanceRings` on `StudentView`. Shows last 30 days of L markers (from the `attendance` prop) + lecture absences (fetched via `getLectureAbsencesForStudent` for admin/teacher, or supplied via `lectureAbsencesProp` for the student portal вАФ same prop-bypass pattern as `attendanceProp`). Hidden when there are zero incidents. `api/student-login.js` was extended to return `lectureAbsences[]` (last 30 days) alongside `attendance[]`.

**Per-month chips on AttendanceRings** вАФ below each monthly donut, up to three clickable chips, each shown only when the count is > 0:

- **Days late: N** (yellow) вАФ from `attendance` `L` rows. Expanded list: `5 Jun ¬Ј 12 Jun ¬Ј 19 Jun` (latest first).
- **Missed Lectures: N** (red) вАФ from `lecture_absences` rows. Expanded list: `19 Jun English ¬Ј 12 Jun Physics ¬Ј 5 Jun Maths`.
- **Missed Exams: N** (darker red) вАФ from `exam_absences` rows bucketed by the exam's date (not `marked_at`). Expanded list: `22 May Mock #5 ¬Ј 8 May Mock #3`.

Expansion model is `expanded: { month, kind } | null` at the component level вАФ single-open across the whole component. Clicking a chip in any month opens it and auto-collapses any other open chip (in the same month OR a different month). Months that have only `L` rows still render a ring at 0%, same applies for months with only lecture/exam misses.

**UI guardrails:**
- Tab strip on Attendance: `Class metrics` (existing content + `LateMarkingWidget` at top, admin-only) / `Lecture log` (new). `LectureLogTab` is shown to admin only; teachers see only Class metrics.
- "Send Morning Late Notifications" disabled until at least one L row exists for the day; "Send Lecture-Miss Notifications" disabled until at least one lecture absence is logged for the (date, batch) pair.
- After send, a fixed-bottom `Alert` shows `sent` / `skipped` counts (or the error). Dismissable.

### Homework / Notes incomplete-work flow (Attendance page, 2026-06-04)

A third Attendance tab (**Homework / Notes**, admin-only) lets faculty flag students with incomplete homework/notes per subject + chapter and alert parents. Structurally a near-clone of the lecture-miss flow, keyed by (subject, chapter, type) instead of (slot_id) and with a resolution stamp the absence flows don't have.

**Data model** вАФ `homework_pending` (see Data persistence). A row is ONE student's incomplete work for one (date, subject, chapter, type). `type вИИ {homework, notes, both}` вАФ the capture form's two checkboxes collapse to a single value (`both` when both ticked). `chapter` is free text (faculty's wording; no syllabus join). `resolved_at` NULL = still open.

**Slice** (`src/store/slices/homeworkSlice.js`):
- `setHomeworkDefaultersForItem(date, subject, chapter, type, lwsIds)` вАФ **reconcile, NOT blind delete-then-insert** (the key divergence from `lectureAbsenceSlice`): selects existing rows for the item, deletes only un-ticked students, inserts only newly-ticked ones, and **leaves remaining rows (and their `resolved_at`) untouched** so resolution survives a card re-edit.
- `resolveHomeworkItem(id)` / `reopenHomeworkItem(id)` вАФ stamp / clear `resolved_at` + `resolved_by`. Row is never deleted.
- `getHomeworkForDate(date)` (cards, filtered to batch students client-side), `getOpenHomeworkForBatch(lwsIds)` (the "Open items" resolve list, unresolved-only across dates), `getHomeworkForStudent(lwsId, sinceDate?)` (RecentIncidents / portal), `markHomeworkNotified(ids)` (stamp `notified_at` after a send).

**Capture UI** (`HomeworkLogTab.jsx`): date + batch pickers вЖТ add-item form (subject `<datalist>` from `timetableMappings`, free-text chapter, Homework/Notes checkboxes) вЖТ per-item cards (`Mark pending` вЖТ `MarkDefaultersModal`, a searchable multi-select modeled on `MarkAbsenteesModal`) вЖТ **Open items** resolve list (per-row `вЬУ Resolved`). Send button is contextual (3 states) keyed on `homeworkSendHistory[\`${date}|${batchName}\`]` and sends only **unresolved** items.

**Send** (`api/send-homework-pending.js`): JWT-gated, sends to student mobile **+ each `parentMobiles[]`** (accountability, same as exam-absence). **One message per (student, item)** вАФ the approved Meta/Wabridge template has a single-item body (`Name/Subject/Topic/Type` fields) with **no date**, so a student with N pending items gets N messages per destination. Positional variables `[name, subject, chapter, typeLabel]` (`{{1}}`=name `{{2}}`=subject `{{3}}`=topic(chapter) `{{4}}`=type; `typeLabel`: `both`вЖТ"Homework and Notes", title-cased). `asciiClean` strips unicode dashes/newlines from the free-text subject + chapter (Meta silent-drop rules вАФ see `feedback_whatsapp_template_param_rules`). The request still carries `date` (used only for the client's send-history key, never a message variable). Env: `WABRIDGE_HOMEWORK_TEMPLATE_ID`. Dev shim in `vite.config.js`. `HomeworkPreviewModal.jsx` mirrors `LectureMissPreviewModal` (editable mobiles, redirect-to test, **pending-only** item-level resend toggle via `notifiedItemKeys`, `bulkUpdateStudentContacts` before send вАФ see "Pending-aware resend"). After send, server `notified_at` is stamped via `markHomeworkNotified` (notified = edits вИТ failedNames) as an audit, separate from the client-side `notifiedItemKeys` that drives the UI.

**Surfacing**: `RecentIncidents` shows **unresolved** homework as `Pending {subject} ¬Ј {chapter}` chips (orange, last 30 days); `api/student-login.js` returns `homeworkPending[]` (12 months) threaded through App.jsx вЖТ StudentView вЖТ RecentIncidents via `homeworkPendingProp` (same prop-bypass as `lectureAbsencesProp`). **AttendanceRings + Monthly Reports surfacing added 2026-06-06** вАФ both count **ALL** homework flagged that month (resolved or not), a factual record like the lecture/exam-miss chips (vs RecentIncidents' open-issues-only view). `AttendanceRings` gets an orange per-month `Homework: N` chip (`homework` prop); `StudentView` fetches homework once via `getHomeworkForStudent` and passes it to both `AttendanceRings` and `RecentIncidents` (deduped). Monthly report PDF gets a `Homework / Notes flagged:` line from `buildMonthlyReport`'s `homeworkFlagged[]` (fed by `monthlyReportSlice.fetchMonthlyReportData`'s 4th bulk read).

### Teacher Feedback (superadmin-only, 2026-06-04)
A new privilege tier (**superadmin**) above admin gates a Teacher Feedback page that aggregates student feedback on each teacher (from the LWS Google Form).

**Superadmin tier (non-invasive).** Superadmin = a Supabase auth user with `user_metadata.role='superadmin'`. They have **no** `teacher` role, so `App.jsx` routes them through `OnlineAdminPortal` (full admin powers). A derived store flag **`isSuperadmin`** (set in `initStore` from the session's role claim; `true` in dev) gates *only* the Feedback page + its sidebar entry вАФ it does **not** touch the `mode==='admin'` checks the rest of the app uses, so superadmin keeps every admin feature. Account `vilas11shinde@gmail.com` was created via SQL (see SECURITY.md).

**WideвЖТlong reshape.** The Form repeats a 9-column block per teacher (8 rating dimensions in fixed order вАФ Clarity, Engagement, Support, Feedback, Pace, Respect, Organization, Availability вАФ + 1 open comment). Teacher identity lives in the form's **section titles, which don't survive into the response sheet**, so the blockвЖТteacher mapping is supplied at import time. Pure helpers in [`src/lib/teacherFeedback.js`](src/lib/teacherFeedback.js): `detectBlockStarts` (counts "Clarity" headers), `reshapeFeedbackMatrix(matrix, teacherNames, {cycle, branch})` вЖТ one row per (response, mapped teacher) вАФ drops unfilled blocks (all-blank) and unmapped blocks; `aggregateFeedback(rows)` вЖТ per-teacher `{n, overall, dims, comments}`; `feedbackTrend(rows)` вЖТ per-teacher overall per cycle; `parseFormTimestamp` (DD/MM/YYYY вЖТ IST ISO).

**Storage + RLS.** `teacher_feedback` table, **superadmin-only RLS** (the project's only role-restricted policy). `teacherFeedbackSlice.js`: `loadTeacherFeedback()` + `importTeacherFeedback(rows)` (session-gated; RLS is the real guard).

**Ingest.** Current data was seeded via SQL (chat-ingest). Future cycles: superadmin uploads the Form export (XLSX/CSV via `parseFeedbackExcel`) in `ImportFeedbackModal` вЖТ maps each block to a teacher (datalist from `timetableTeachers`) вЖТ sets a `cycle` label + `branch` вЖТ preview count вЖТ import. Teacher names map to canonical `timetableTeachers` entries (e.g. **Akash Rathod Sir**, distinct from Akash Singh Sir).

### Daily Quiz (2026-06-05)
In-app daily MCQ quizzes вАФ a **tracked replacement for the Google Forms** a teacher used. The win over Forms: students log in by mobile (вЖТ `lws_id`), so every attempt attributes cleanly with **zero name-matching** (no dedup mess), and questions carry chapter/difficulty tags so results feed analytics.

**Tables (deliberately SEPARATE from `exams`)** вАФ so daily quizzes never pollute mock dashboards / Toppers / projected-score, and never trip the exam-absence WhatsApp flow:
- `quizzes (id uuid, title, subject, batch, branch, marking jsonb default {correct:1,wrong:0}, questions jsonb, opens_at, closes_at, status text CHECK in ('draft','published'), created_by, created_at, updated_at)`. `questions[]` mirrors `exams.questions[]` (q, question, optionAвАУD, answer, chapter, subtopic, difficulty, solution) so all existing rendering reuses. `batch` is comma-joined for multi-batch (read via `getExamBatches`); empty batch = all-batches.
- `quiz_attempts (id uuid, quiz_id FK CASCADE, lws_id FKвЖТstudents CASCADE, student_name, answers jsonb {qNum:'A'..'D'}, score, correct, incorrect, not_attempted, started_at, submitted_at, UNIQUE(quiz_id, lws_id))`. Stores the chosen **letter** (not 1/-1/0) so review + recompute work. RLS: both `authenticated` (admin **and teacher**).

**Persistence** mirrors exams: quizzes live in their own table, **stripped from the `faculty_state` JSONB** in `saveToSupabase`, kept in `saveToStorage`'s allow-list for dev-disk persistence, loaded via `loadQuizzesFromSupabase()` (called in `initStore` admin path + `TeacherPortal` mount; stale `quizzes` stripped from the JSONB on load). Store key: **`quizzes`** (DEFAULTS `[]`).

**Authoring (teacher + admin).** Teachers are read-only by UI convention only вАФ at the DB level they're authenticated, so RLS already permits quiz writes; exposing the **Quizzes page** + sidebar entry to teacher mode is the whole "teacher authors directly" feature (no new RLS). `src/pages/Quizzes/` вАФ `index.jsx` (list + Copy-link + Results), `QuizEditor.jsx` (manual MCQ entry + batch picker + close-time + publish). **PYQ-bank picker + Excel/paste import were chosen but deferred** (manual-entry unblocks daily use).
- Slice `quizSlice.js`: `addQuiz/updateQuiz/deleteQuiz` (dual-path) + read `getQuizAttempts(quizId)` / `getQuizAttemptsForStudent(lwsId)` (session-gated). **Key divergence from examsSlice: `_save()` is called ONLY in the no-session (dev) branch** вАФ quizzes write to their own table when a session exists; calling `_save()` then would round-trip the whole `faculty_state` blob and a quiz-authoring *teacher* could clobber admin data. `quizSupabase.js`: `buildQuizRow`, `upsertQuiz`, `deleteQuizById`.
- Pure libs: `src/lib/quiz.js` (`gradeQuizAttempt`, `quizStatus`, `validateQuizForPublish`, `quizQuestionComplete`, `stripAnswerKey`, `LETTERS`); `src/lib/quizStats.js` (`quizCohort`, `quizSummary`, `quizQuestionStats`, `quizNotAttempted`).

**Student taking (prod-only, like `student-login` вАФ no dev shim).** `api/student-quizzes.js` POST `{mobile}` вЖТ batch-filtered open quizzes with the **answer key stripped** (state `open`) + already-attempted ones with key + answers for review (state `done`); also returns `name`. `api/quiz-submit.js` POST `{mobile, quizId, answers}` вЖТ **grades server-side** against the key the student never received, enforces the close-window + one-attempt (UNIQUE + pre-check), writes the attempt, returns the graded review. Both reuse `gradeQuizAttempt`/`stripAnswerKey` via `import '../src/lib/quiz.js'` (resolves in Node test env + Vercel bundle). **Integrity: the answer key only reaches the browser AFTER submit.**

**Shareable link.** Each published quiz has a **рЯФЧ Copy link** button вЖТ `${origin}/?quiz=<id>`. `App.jsx` handles `?quiz=<id>` as an early, store/auth-independent route вЖТ `QuizLinkPage.jsx`: a focused single-quiz page with a **one-time mobile entry** (remembered via the existing `nda_student_session` or a lightweight `nda_quiz_mobile` key) вЖТ taker вЖТ submit вЖТ review. Identity is still required (so attempts attribute) but there's no portal chrome. Shared taker/review live in `src/pages/Quizzes/quizTaking.jsx` (used by both `StudentQuizzes` and `QuizLinkPage`).

**Surfacing.** Student portal: `StudentQuizzes` ("рЯУЭ Daily Quiz" section atop `StudentPortal`, `mobile` from session). Admin/teacher: per-quiz **`QuizResults`** dashboard (attempts/cohort, avg %, per-question correct % via `HeatBar`, who-hasn't-attempted as red chips) reached via the Results button; `StudentQuizHistory` card in `StudentView` (admin/teacher only). `quiz_attempts` reads are session-gated, so these populate only in prod with real attempts.

### Monthly Reports (2026-05-24)
Admin-only sidebar entry that generates per-student PDF report cards for a (batch, month) pair. Compute-on-demand вАФ no persistence. Designed for parents.

**Page** [`src/pages/MonthlyReports/index.jsx`](src/pages/MonthlyReports/index.jsx): month picker (default = previous calendar month) + batch picker (from `syllabusBatches`). "Generate" calls `fetchMonthlyReportData(month, lwsIds)` (one Supabase query per table for the whole cohort, not three √Ч N round-trips). After generating, renders one `ReportRow` per cohort student. Per-row Download. Bulk **"Download all as ZIP"** packages every report into one archive via JSZip (loaded dynamically).

**Cohort rule** in `getMonthlyReportCohort(profiles, batch, month)`: Active `accountStatus` + `batches[]` includes selected batch + `regDate вЙ§ last-day-of-month`. Skips variant-keyed entries. Students with zero exams still get a report (attendance only).

**Builder** [`src/lib/monthlyReportBuilder.js`](src/lib/monthlyReportBuilder.js): pure `buildMonthlyReport({ profile, month, exams, attendance, lectureAbsences, examAbsences, batchChapterTimelines, syllabusPrograms })` returns sections object вАФ `meta`, `examTable` (with ABSENT rows from exam_absences, regDate-filtered), `attendance` (P/A/L counts + percentage + missed lectures list + late dates), `nextMonthFocus` (chapters scheduled for following month from `batchChapterTimelines`).

**PDF** [`src/lib/monthlyReportPdf.js`](src/lib/monthlyReportPdf.js): jsPDF + jspdf-autotable (dynamic imports, same pattern as `examPdf.js`). Renders REPORT CARD title + underline, Name/Month meta block (no Roll No), exam table mirroring the institute's paper format (ABSENT in danger colour), attendance row with conditional non-zero segments via `attendanceDescriptor(a)`, optional faculty remark (transient, typed at preview time), next-month focus footer, institute footer line.

**Bulk ZIP** [`src/lib/monthlyReportZip.js`](src/lib/monthlyReportZip.js): `buildMonthlyReportsZipBlob(items)` iterates the cohort, renders each PDF sequentially, writes to a JSZip archive. `zipFilename(batch, monthLabel)` sanitises to `{batch}_{Month}_Reports.zip` (no parens, no whitespace).

**Remark UX**: typed at generate-time, lives in page-level state as `{ [lwsId]: remark }`, written into the PDF on download. Not persisted вАФ re-typed if faculty re-generates. Per the agreed compute-on-demand decision.

**Trimmed from the original spec (2026-05-24, faculty feedback)**: Subject summary table + Weakest chapter line removed (rendered the trend вЖС/вЖУ glyphs as garbage via jsPDF's WinAnsi encoding вАФ same class as the LaTeX-symbols bug in `examPdf.js`). ### Mentorship daily nudge (2026-06-19)
A coverage-rotation tool for the mentorship program: each mentor teacher is allocated a set of students, and every **weekday morning** WhatsApp sends the mentor **3 of their mentees** to personally check in with (talk to them, review homework/notes). It is **not** an at-risk alert вАФ it's a load-spreader so every mentee gets regular 1:1 attention, picked **randomly without replacement** until the whole roster is covered, then a fresh round. **The only WhatsApp flow whose recipient is a teacher, not students/parents.**

**Tables** (see [`DATABASE_SCHEMA.md`](./DATABASE_SCHEMA.md) ¬І10): `mentor_assignments` (lws_id PK вЖТ teacher_id; one mentor per student) + `mentor_nudges` (append-only event log). The log **doubles as the rotation cursor** вАФ no stored queue. See [[feedback_event_log_over_derive]].

**Rotation** вАФ pure `pickDailyMentees(mentees, nudgeLog, {n:3, today})` in [`src/lib/mentorNudge.js`](src/lib/mentorNudge.js): rank the mentor's active mentees by past nudge **count**, take the **lowest-count tier only** (random tiebreak). This gives clean rounds вАФ the last day of a round may carry **fewer than 3** names rather than pulling next-round students forward. Idempotent + resumable within a day (excludes mentees already nudged `today`). `fmtNudgeDate` ("22nd June 2026"), `isNudgeDay` (MonвАУFri), `istDateString` (IST send-day from a UTC instant) round out the lib.

**Send** [`api/send-mentor-nudges.js`](api/send-mentor-nudges.js): **dual-auth** вАФ Vercel cron (GET, `Authorization: Bearer <CRON_SECRET>`, scheduled `0 2 * * 1-5` in `vercel.json` = 07:30 IST) for the real run, OR admin JWT (POST) for `{dryRun, redirectTo, force}`. Reads via the **service-role** client (the cron has no user session): `mentor_assignments` + `students` (active-only filter via live `account_status` вАФ Block/inactive mentees are skipped, self-healing) + `mentor_nudges` history + `faculty_state.timetableTeachers` for the mentor's `mobile`. Builds Wabridge vars `[fmtNudgeDate(today), names.join(', ')]` (template id env `WABRIDGE_MENTOR_NUDGE_TEMPLATE_ID`; no teacher-name var вАФ body greets "Dear Teacher"). **`mentor_nudges` rows are inserted ONLY after a successful real send** вАФ so a failed send doesn't burn a rotation slot, and a `redirectTo` test send or `dryRun` never advances the rotation. Dry-run works even before the template env is set. Dev shim in `vite.config.js` (POST path only).

**Admin UI** вАФ Settings вЖТ **Mentorship** ([`src/pages/Settings/MentorshipTab.jsx`](src/pages/Settings/MentorshipTab.jsx)):
- **Preview today's picks** (POST `{dryRun, force}`) вАФ shows each mentor's would-be 2 mentees; non-destructive; `force` bypasses the MonвАУFri gate for any-day preview.
- **Send test to [mobile]** (POST `{redirectTo, force}`) вАФ sends the real message to one number; non-destructive (no rotation advance).
- **Mentee assignments** (`MenteeAssignments`, via `mentorSlice` `fetchMentorAssignments`/`setMentorAssignment`/`removeMentorAssignment`): reassign a mentee's mentor, remove from mentorship, and a highlighted **"active students with no mentor"** list so nobody silently falls out of the rotation. Teacher mobiles are entered in Settings вЖТ Teachers (the `timetableTeachers[].mobile` field).

**Rollout dependency**: the cron is fail-closed вАФ until `CRON_SECRET` is set in Vercel the scheduled call is rejected (no accidental sends). First real send also needs `WABRIDGE_MENTOR_NUDGE_TEMPLATE_ID`, teacher mobiles, and a live test to confirm the `[date, students]` variable order (per the template-param rules вАФ positional order isn't knowable from the template ID).

### Hostel & Mess (APJ boarders, Phase 1вАУ2 вАФ 2026-07-08)
Boarder attendance across the hostel/mess day, scoped to `branch='APJ'`. **Admin-only** tab in the Attendance page ([`src/pages/Attendance/HostelTab.jsx`](src/pages/Attendance/HostelTab.jsx)). **Exception-capture** (default-present) вАФ the same model as the lecture log. Purpose: student safety/whereabouts + discipline. See [[project_hostel_attendance]].

**Checkpoints** вАФ 5 capturable: `hostel_am`, `breakfast`, `lunch`, `dinner`, `hostel_pm`. The chain view also shows a **`class`** column *derived* from `student_attendance` (never captured here); lectures are out of the Phase-1 chain.

**Mark view** вАФ pick a date + checkpoint; the APJ boarder roster loads all-green (Present). Tapping a boarder cycles Present вЖТ Absent вЖТ Sick вЖТ Out-pass вЖТ Present. Boarders with an active `leaves` row show a locked **Leave** state (pre-explained, not tappable). **Save** does a delete-then-insert of the exception set for that (date, checkpoint) via `setCheckpointExceptions` вАФ no row = present. **Batch + boys/girls filters** narrow the *list only* (a display lens over 150+ boarders) вАФ the `edits` set, the Roster/Expected tally, and reconciliation stay whole-hostel, so filtering to one wing and saving never drops another wing's marks. Branch filter auto-appears only if a second hostel branch exists (`HOSTEL_BRANCHES`, today `['APJ']`).

**Reconciliation gate** вАФ the two roll checkpoints (`hostel_am`/`hostel_pm`) add a headcount gate. Expected-in-dorm = roster вИТ *away* (away = absent + out-pass; sick counts as in-dorm). The warden enters the physical headcount and closes the roll: `confirmRoll` writes `checkpoint_confirmations` with `reconciled = (headcount == expected вИТ away)`. A mismatch is persisted as **`reconciled=false` вАФ an OPEN incident** the Chain view surfaces. Save exceptions before reconciling (the button is gated on a clean save).

**Chain view** вАФ `buildDailyChain` ([`src/lib/analytics/chain.js`](src/lib/analytics/chain.js), pure) composes hostel/mess exceptions + derived class attendance + active leaves into a per-boarder timeline. An **anomaly** = a checkpoint `status='absent'` with no covering leave вЖТ the boarder "fell off the chain"; `firstBreak` marks where. The board lists anomalies (and open unreconciled rolls) for the day вАФ the same-day safety signal paper registers can't give.

**Leaves** = the honesty mechanism: an active leave/out-pass overlapping a day explains **every** checkpoint that day (day-granular; `resolveOnLeave` does the overlap test), so those gaps aren't anomalies. `leavesSlice` (`addLeave`/`getActiveLeaves`/`deleteLeave`).

**Warden alert (Phase 2)** вАФ the Chain view has an **Alert warden** button + an inline warden-number editor (`hostelAlertMobiles[]`, config). Pressing it POSTs `{date}` to [`api/send-hostel-alert.js`](api/send-hostel-alert.js), which вАФ with the service-role client вАФ **re-loads roster + attendance + checkpoints + leaves and re-computes the chain server-side** (never trusts a client-sent list), then WhatsApps the unexplained list (`buildWardenAlert` вЖТ vars `[date, "Name - Checkpoint, вА¶"]`) to each warden number. Admin-JWT (cron-secret branch wired but **not scheduled**). **Fail-closed** on missing `WABRIDGE_HOSTEL_ALERT_TEMPLATE_ID` (dry run previews without it); only fires when count>0 and a number is configured; stateless (no send log yet). Var order + template are **provisional until a live redirect-to test** вАФ see [[reference_whatsapp_templates]] + [[feedback_whatsapp_template_param_rules]].

**Deferred (Phase 3+):** nightly cron auto-send, a persistent alert/notify log ("did we alert?"), parent notification, per-student boarding timeline in `StudentView`, compliance % reports, the day-scholar `residential` split, time-granular partial leave.
