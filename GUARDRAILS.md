# NDA Maths Tracker тАФ What not to change (guardrails)

> Extracted from [`CLAUDE.md`](./CLAUDE.md) on 2026-06-09 to keep that file lean. These are the behavioural invariants the codebase depends on тАФ each is something that broke something else when changed, or that prevents a class of bugs from re-appearing. The *why* trail for any of them is in [`DECISIONS.md`](./DECISIONS.md); column-level schema in [`DATABASE_SCHEMA.md`](./DATABASE_SCHEMA.md). New guardrails go here, not in CLAUDE.md.

- Do not persist `apiKey` anywhere тАФ memory only.
- Subject filter state is local per page тАФ do not lift to Zustand.
- `StudentView` subject filtering is self-contained тАФ no prop threading.
- Use `useMode()` for visibility тАФ no new `IS_READ_ONLY` imports in components.
- `ModeContext` default is `'admin'` тАФ changing it breaks tests.
- All hooks must be called **before** any early returns in page components.
- Do not filter batch dropdowns or exam lists on `exam.batch`/`exam.branch` directly тАФ always use `getBatchOptions` / `getExamsForBatch` / `getExamsForBranch` from `src/lib/analytics`.
- Toppers + Dashboard batch/branch filters are **student-centric (current members)**: they intersect the analytics `validNames` set with `getBatchMemberNames` / `getBranchMemberNames`. Do NOT regress to ranking *all attendees* of a batch's exams тАФ that leaks cross-cohort co-attendees (and was the reported "batch/student mismatch"). Do NOT scope the branch filter on the `exam.branch` tag for these surfaces тАФ it would truncate a moved student's earlier-branch history; use `getExamsForBranch` (roster) + the `validNames` intersection. Branch and batch are kept in sync per student (`students.branch` тЖФ the batch's branch), so both filters key off current membership. See [[feedback_current_cohort_over_exam_roster]].
- Dashboard score widgets use **%-of-max**, never raw `totalMarks` averaging тАФ exams have different paper sizes, so a raw mean is meaningless (that was the old "Avg Score" bug). Do not reintroduce a raw-totalMarks class average.
- Chapter-name tag validation (`ValidationIssuesPanel`, used by `Step1Upload` + `ReuploadTagsModal`) is a non-blocking **WARNING** тАФ do NOT re-add a hard block on unrecognised chapters. The check validates against the NDA list (`getValidChapters`), which is wrong for non-NDA papers (school Class-10 tests) whose chapters legitimately aren't in it тАФ blocking them left no way to upload. Keep the amber panel + the accept-suggestion ("Did you meanтАж?") affordances, but the Proceed/Next button must stay enabled. The **GAT** per-question-`Subject` requirement (`validateGatSubjects` + the GAT `hasSubjectColumn` check) DOES stay blocking тАФ that's routing, not chapter names. See DECISIONS.md (2026-06-15).
- **Every %-of-max site MUST compute the denominator via `examMaxMarks(exam)`** (`src/lib/analyticsHelpers.js`, re-exported from `lib/analytics`), NOT inline `questions.length ├Ч marking.correct`. The helper prefers an explicit positive `exam.maxMarks` (offline exams have `questions: []`, so the inline form returns 0 and the exam silently vanishes from trends/Toppers/% columns). Current callers: `examAvgPct`, `getToppers`, `computeConsistency`, `getExamTopStudents`/`getExamBottomStudents`, `buildMonthlyReport`, `examPdf`, `Exams.jsx`, `ExamInsightsPanel`, `StudentView`. Add new score surfaces through the helper. See **Excel upload format тЖТ Offline results**.
- Offline exams are derived via `!exam.questions?.length` (no stored flag). Do NOT add per-question consumers that assume `questions[]` is non-empty without guarding тАФ they correctly render empty for offline exams; surface an "Offline" notice, never misleading zeros. `addExam(exam, { syncAbsences })` defaults `syncAbsences:true` (Evalbee path unchanged); the offline modal passes `false` by default so a totals upload doesn't WhatsApp parents.
- `getPriorityChapters` priority is `weightPct├Ч(1тИТaccuracy)` with untested chapters using accuracy 0 (so a high-weight untested chapter still surfaces, flagged `tested:false`). `getClassProjectedAvg` reuses `getToppers(тАж, 0, тАж)` for identical regDate scoping тАФ do not fork a parallel projection path.
- `FrequencyTableEditor` is rendered ONLY by `Settings/NdaWeightageTab` (moved off the Dashboard 2026-06-01). Do not re-add it to the Dashboard тАФ NDA weightage is config, and Settings is the config surface. The file stays under `pages/Dashboard/` for git-history continuity; that's intentional, not a leftover.
- `AttendanceRollup` is **class-wide for one day** and intentionally does NOT read the Dashboard's subject/branch/batch/exam filter chain тАФ attendance isn't exam-scoped, and faculty want every branch visible at once. Do not wire the page filters into it. `buildAttendanceRollup` bucketing is **Absent = status `A`, Present = everyone else** (P / L / `-` / no record = enrolled тИТ absent) тАФ do NOT add a separate Late column or reclassify no-record as absent. Cohort is **Active-only** (skips non-Active + variant-keyed + batch-less profiles, same guards as `getExamAbsentees`); a multi-batch student is counted under each batch by design. Branch grouping is `syllabusBatchBranches[batch] || profile.branch` тАФ keep the fallback so unmapped batches still appear. Batches sort by `batchStdRank` (class number before "th", non-numbered programs last) тАФ do NOT revert to plain `.sort()`. Each branch table has a **Total footer row** (`idBatch='__total__'`) that sums its batches and is itself drill-downable; total = ╬г batch counts (a multi-batch student counts once per batch, so branch totals can exceed headcount тАФ intentional, consistent with the per-batch cells). `fetchDailyAttendance(null)` resolves to the latest recorded date; do not cache attendance in the store (same rule as the rest of the app тАФ always fetch on demand).
- Syllabus Tracker batch names are independent of **student-profile** batch names (`profile.batches[]`) тАФ do not derive them from `studentProfiles`. As of 2026-05-20 syllabus batch names match `timetables[].batchName` 1:1 (unified scheme). Use the unified `renameBatch(old, new)` / `deleteBatch(name)` actions on the configSlice when an admin renames or deletes a batch тАФ they delegate to both `renameSyllabusBatch` + `renameTimetableBatch` (and respectively `deleteSyllabusBatch` + the timetable/exam-schedule guard) so the two stores can't drift. The side-specific actions (`renameSyllabusBatch`, `renameTimetableBatch`, `deleteSyllabusBatch`) are still exposed and still used internally by the unified actions and by code that legitimately needs to touch one side (e.g. `migrate_unify_batches.js`); UI code should always go through the unified action.
- Syllabus branch filter branches are sourced from `timetables[].branch` тАФ do not create a separate branch list for the Syllabus Tracker. `syllabusBatchBranches` cascades on batch rename/delete.
- To rename a timetable's `batchName`, use `renameTimetableBatch(oldName, newName)` тАФ it cascades to `examSchedules[].batchName`. Do not use `updateTimetable(id, { batchName })` for renames; that's a bare patch that leaves exam schedules pointing at a stale name.
- `updateTimetableTeacher(id, patch)` takes `{ name?, email? }` тАФ do not pass a bare string (breaking change from original signature).
- Exam Schedule `branch`/`batchName` must come from `timetables[]` entries тАФ do not derive from `syllabusBatches` or `exams[].batch`.
- `guardian_mobile` from Excel merges (appends if absent) into `parent_mobiles[]` тАФ do not change to overwrite semantics; manually-added parent numbers must survive re-import.
- Touch targets must be тЙе 44px on all mobile-facing screens (`min-h-[44px]` or equivalent padding). Use `py-2.5`+ for buttons, `py-3`+ for nav items.
- `findDuplicateCandidates` with no `branchFilter` does a flat cross-branch scan тАФ do not revert to per-branch-group iteration.
- The `name_subset` signal requires `shorter.length >= 2` тАФ do not remove this guard (prevents false positives on shared single-word surnames).
- `name_token_edit` requires `min(unique-token length) >= 5` (`TOKEN_EDIT_MIN_LEN`) тАФ do not lower without expecting Anil/Sunil/Amit-class false positives. Levenshtein cap is 2 (`TOKEN_EDIT_MAX_DIST`). Order-independent тАФ both `"Neel Vardhamane" / "Neel Wardhamane"` and the reverse fire the same way. Patil/Patel surfacing as a candidate is INTENTIONAL (documented trade-off) тАФ never auto-link; faculty Skips. Don't add per-name carve-outs for known distinct surnames.
- `name_token_prefix` requires `singleToken.length >= 4` (`TOKEN_PREFIX_MIN_LEN`) and `profileTokens.length >= 2`. The first guard protects 3-letter `Anu/Raj/Om` from matching every Anu* profile; the second prevents the rule from firing on `"Rajivkumar"` vs `"Rajivkumar"` (true duplicates handled upstream by `getUnmatchedExamNames`).
- `name_initial_match` requires both sides тЙе 2 tokens, longer side тЙе 3 tokens (real first+last anchor тАФ prevents `V. Sharma` тЖФ `Vijay Sharma` siblings-class noise), exactly one unique token per side, one unique matches `/^[a-z]\.?$/`, and the other unique starts with that initial. Order-independent (initial may be on either side). Closes the middle-initial gap where `name_subset` rejects on token inequality (`v.` тЙа `vijay`), `name_token_edit` rejects on the 5-char floor (`"v."` is 2 chars), and Jaccard falls below 0.75 due to punctuation bigrams. Surfaces `Anant V. Sharma` тЖФ `Anant Vinay Sharma` too тАФ that's the ambiguous case faculty should review; do NOT add per-letter carve-outs.
- `addNameVariant` deduplicates before appending тАФ do not change to unconditional push.
- `addNameVariant` (Supabase path only) calls `cleanStaleAbsencesForVariant(supabase, lwsId, variantName)` after the variant is persisted but before `refreshStudents`. This sweeps `exam_absences` rows for the student where `exam_results` already has the just-linked variant as an attendee тАФ pre-link audit rows that are now factually wrong (the student attended under a spelling the system didn't yet recognise). Do NOT remove тАФ without it, every Find-Duplicates тЖТ Link-as-variant action leaves a stale "Missed exam" chip on the student's RecentIncidents + MissedExams surfaces.
- Search list shows only canonical names тАФ do not add variant names (keys where key тЙа `profile.name`) back into the search list; they would create duplicate entries for the same student.
- `StudentView` normalizes exam records in-memory via `normalizedExams` тАФ do not remove this or revert `getStudentExams` to using raw `exams`; doing so breaks analytics for students whose exam records are stored under a variant spelling.
- Timetable Excel export uses `xlsx-js-style` тАФ do not change the import back to `xlsx`.
- `batchChapterTimelines` cascades on `deleteProgram`, `deleteSubject`, `deleteChapter`, `renameSyllabusBatch`, `deleteSyllabusBatch` тАФ do not remove these cascade blocks.
- `clearSubjectProgress` intentionally does NOT clear `batchChapterTimelines` тАФ the planned chapter schedule must survive a status reset.
- The "Timeline" column in `SubjectAccordion` is a fixed column (not a user-defined tracking column). If a user has a tracking column also named "Timeline", they will see two Timeline columns тАФ the old one should be deleted from the program's tracking columns.
- `stripLatex` in `examPdf.js` must only output ASCII + safe Latin-1 (`├Ч├╖┬▒┬╖`). Do not re-introduce Unicode math symbols (Greek, set ops, arrows, `тЙдтЙетЙа`, `тИИтИктИй`, `тДЭ`) тАФ all fall outside jsPDF Helvetica's WinAnsi encoding and render as garbage.
- The KaTeX render component is named `Math` (`src/components/ui/Math`). Importing it as `{ Math }` **shadows the global `Math`** тАФ any `Math.round`/`Math.max`/etc. in that file then throws "Math.round is not a function". In a file that needs both, alias the import (`import { Math as Tex } from 'тАж/Math'`) тАФ `QuizResults.jsx` does this. Pure-presentation files (e.g. `quizTaking.jsx`) keep the bare `Math` import since they do no arithmetic.
- `saveToSupabase` must gate on `supabase.auth.getSession()` before writing тАФ do not remove this check. Teacher and student visits must never overwrite admin data.
- Do not use `.catch()` on Supabase query builder chains тАФ `PostgrestFilterBuilder` is a thenable, not a full Promise; `.catch` is `undefined` and throws silently. Always use `async/await` with `{ error }` destructuring inside the `.then()` callback.
- `studentSlice.js` mutations must check `getSession()` before choosing Supabase vs fetch path тАФ do not remove this dual-path logic. Dev mode depends on the fetch path; Vercel depends on the Supabase path.
- `loadStudentsFromSupabase()` must be called after `faculty_state` loads in `initStore()` тАФ it overwrites the stale `studentProfiles` baked into `faculty_state` with fresh data from the normalised tables. Do not remove this call.
- `loadInsightsFromSupabase()` must be called in `initStore()` after `faculty_state` loads (admin path only) тАФ same reason as `loadStudentsFromSupabase`: it overwrites the stale `savedInsights` baked into the JSONB blob with fresh data from `class_reports` / `student_plans`. Do not remove this call.
- `saveToSupabase` strips both `exams` AND `savedInsights` from the JSONB blob before writing тАФ do not remove either field from the destructure. Re-introducing them would double-write the data and let the JSONB drift out of sync with the normalised tables on admin mutations.
- `insightsSlice.js` mutations are append-only тАФ `saveClassReport` / `saveStudentPlan` insert new rows each time, never update in place. Do not change to upsert; history would be lost.
- `class_reports` and `student_plans` are read by `loadInsightsFromSupabase()` which collapses to "latest per scope" тАФ preserves the legacy `{ classReport, studentPlans }` store shape. Do not change the in-store shape; the Insights page depends on it.
- `loadExamsFromSupabase()` is called in `initStore()` (admin) and in `TeacherPortal` mount (teacher) тАФ both paths must await it; removing it leaves the store with stale JSONB data (or no data).
- `fetchAllRows` pagination in `persist.js` is required for all Supabase tables that can exceed 1000 rows (`exam_results` already at 1472+). Do not replace with a bare `.select('*')` тАФ it silently truncates.
- `migrate_exams_to_supabase.js --cleanup` only prints the cleanup SQL; it does not execute it. Run the SQL manually in Supabase SQL editor only after confirming exam counts match and the live app shows correct data. Never run cleanup before seeding.
- `students_db.exams[]` must not be seeded into Supabase тАФ confirmed dead data (no code path reads it). The normalised schema deliberately omits it.
- `exam_results` marks (`total_marks`/`correct`/`incorrect`/`responses`) are **Evalbee-graded at upload**, not derived from `exams.questions[].answer`. Do NOT attempt to re-grade results against a corrected answer key from Supabase alone тАФ `responses` stores only the `1/-1/0` verdict, never the student's chosen option, so credit for students who picked the now-correct option is unrecoverable from the DB. A true re-grade needs the original Evalbee export's `Q N Options` columns (re-grade choices against the new key) or fixing the key inside Evalbee + re-upload. To bulk-fix a key in `exams.questions` JSONB use `jsonb_agg(CASE (elem->>'q')::int WHEN n THEN jsonb_set(elem,'{answer}','"X"') тАж END ORDER BY ord)` over `jsonb_array_elements(questions) WITH ORDINALITY` (preserves order + length). See [[reference_exam_grading_data_model]].
- `student_batches` PK is `(lws_id, batch_name)` тАФ to rename a batch, you must DELETE old rows and INSERT new ones; you cannot UPDATE the PK in place.
- `student_attendance` UNIQUE constraint is `(lws_id, date)` тАФ batch column was dropped (2026-05-07); upsert conflict target is `lws_id,date`.
- Vercel env vars `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` must be set in the Vercel dashboard; they are not in the repo. `src/lib/supabase.js` returns `null` when they are absent (safe no-op on dev/GH Pages).
- Attendance data is not stored in Zustand тАФ always fetched from Supabase on demand. Do not add it to the store or persist it to `faculty-data.json`.
- `attendanceSlice.importAttendance` matches students by mobile first, then by name тАФ do not change to name-only matching; mobile is more reliable when names vary across XLS files.
- `AttendanceRings` uses `relative` + `absolute inset-0` overlay for the % label тАФ do not revert to the `marginTop` hack; it places text outside the ring bounds on light backgrounds.
- `student_logins` insert uses `.then(() => {})` тАФ do not add `await`; it must remain fire-and-forget so login latency is unaffected.
- `AttendanceRings` renders months latest-first (`b.localeCompare(a)`) тАФ do not revert to ascending sort.
- `buildConsecutiveAbsent` measures against the **global** non-Sunday date sequence (all students share the same reference dates тАФ do not switch to per-student date filtering) AND reports the TRUE streak length by walking backwards from the latest known date until the first `P` / `L` / missing record. `since` is the earliest `A` in the actual streak, NOT `earliest-of-last-N`. Do not revert to "check only last N dates" тАФ a student absent for 10 consecutive days must show `since 10 days ago`, not `since N days ago`. Each result carries `count` (the streak length, already computed internally тАФ do not recompute downstream) and the array is sorted by `count` desc with name-asc tiebreak; do not revert to alphabetical-only sort.
- `StudentView`'s `attendance: attendanceProp = null` prop bypasses the Supabase fetch when provided тАФ required for the student portal because students have no Supabase auth session (RLS blocks unauthenticated reads). Removing the prop bypass breaks attendance rings in the student portal.
- Attendance page has no batch filter by design тАФ consecutive absence detection is class-wide.
- `FindDuplicatesTab` must scan `studentList` (raw Supabase array), not `Object.values(studentProfiles)`. The map collapses identical canonical names; the scan needs to see both entries to flag them.
- `StudentView`'s `!examData.length` early return must render the subject selector + ProfileCard. Removing the selector strands GAT-only students on the default Maths filter with no escape.
- `importStudentsDB` must call `set({ studentProfiles, studentList: students })` together тАФ the two are kept in sync and both are required (profiles for lookups, list for duplicate scanning).
- Do not use `import.meta.env.DEV` anywhere in `src/`. Vercel's Vite 8.0.3 substitutes it with `true` in prod builds (confirmed 2026-05-20). Use `IS_READ_ONLY` from `src/config.js` (or `IS_DEV = !IS_READ_ONLY` in `persist.js`) for any dev-vs-prod branching. Other `import.meta.env.*` (`BASE_URL`, `VITE_SUPABASE_URL`) appear unaffected.
- Do not remove the `stream` тЖТ `src/stubs/empty.js` alias in `vite.config.js`. xlsx's optional stream-probe will crash the bundle at startup without it. If xlsx ever drops the probe (or moves it behind a feature flag), the alias can go тАФ verify by rebuilding and checking that `(va = HM())` no longer appears near `.Readable` in the minified output.
- `useImportFlow.handleStudentFile` must call `loadExistingStudents()` (not a bare `fetch('/api/students-db')`) before `mergeStudents`. The bare fetch 404s on Vercel and causes silent duplicate creation. Same rule for any future code that needs the snake_case students baseline тАФ go through the helper so the Supabase path is taken when an admin session exists.
- `mergeStudents` returns `conflicts: []` always (even when empty). Do not remove this field or change to `null` тАФ `ImportStudentsModal.jsx` reads `mergeResult.conflicts?.length` and the test `mergeStudents тАФ return shape > always includes conflicts in the result` asserts on it.
- Tiered match's "exactly one hit" rule in steps 2 and 3 is a hard requirement тАФ do not change to "first hit wins". Ambiguous matches must surface in `conflicts[]` so admin resolves manually; auto-picking risks attaching exam/attendance/fee history to the wrong student.
- Step 3 (name+branch) requires a non-empty branch on BOTH sides тАФ do not relax to name-only matching. Two students named "Rohan Patil" with no branch would otherwise auto-merge incorrectly.
- Tiered match still skips inserting when EIS is blank AND no match is found via steps 2/3 тАФ do not change to insert-on-blank-EIS. Creating EIS-less rows leaves future re-imports unable to match the student and breaks the canonical identifier contract.
- `mergeStudents(_, _, { defaultBranch })` is "fill blanks only" тАФ applies to new inserts and existing students whose branch is empty. Never overwrites a non-empty branch. Do not flip this to "overwrite all" without adding an explicit confirm step in the UI; a wrong dropdown pick would clobber correctly-aligned data.
- The Students page renders either the table OR the StudentView тАФ never both. Do not revert to the stacked "Pattern X" layout without explicit user request; it was deliberately removed 2026-05-20.
- `deleteStudent` must always go through a `window.confirm` (or stricter) at the UI layer. Do not bypass тАФ the action is irreversible at the attendance/login level. The confirm message must name the student and mention what data is lost.
- `deleteStudent` clears `activeStudent` if (and only if) it matched the deleted student's name. Do not unconditionally clear; switching to another student before deleting should not jump back to the table.
- The privileged role is `'admin'`, not `'faculty'`. Do not reintroduce `mode === 'faculty'` or `<ModeContext.Provider value="faculty">` тАФ use `'admin'`. The Supabase table name `faculty_state` and the dev file `data/faculty-data.json` are deliberately retained (internal storage, no UI). The user-facing copy "LWS Pune faculty" in the login footer and message templates is also retained тАФ it refers to the real-world teaching staff.
- The login page has exactly two tabs (Student / Admin ┬╖ Teacher). Do not split the staff tab back into separate Admin and Teacher tabs тАФ they share a single Supabase auth form, and the role is decided server-side from `user_metadata.role`.
- `branches[]` is owned by `configSlice` and seeded by `seedBranches(saved)` on first load (union of `timetables[].branch` тИк `Object.values(syllabusBatchBranches)`). Do not derive a branch list inline in components going forward тАФ read `useStore(s => s.branches)`. Existing inline `[...new Set(timetables.map(t => t.branch))]` patterns in `Dashboard`, `Exams`, `Toppers`, `AddExamScheduleModal`, `ExamScheduleView`, `TimetablePage` were left in place because they're cosmetic-only (filter dropdowns); migrate them opportunistically, but do not introduce new inline derivations. `SyllabusPage` was already migrated to `useStore(s => s.branches)`.
- `renameBranch` (configSlice) cascades to `timetables[].branch`, `examSchedules[].branch`, and `syllabusBatchBranches` values. It does NOT touch `students.branch` or `exams.branch` тАФ those were aligned to the central namespace via one-off SQL on 2026-05-21 (see decisions log). The Evalbee XLS does not carry per-student branch info (no `Branch` column in the documented format), so `mergeStudents` won't undo the alignment on re-import. If branch values ever drift again, run the same SQL pattern тАФ there is no UI surface for this rename.
- `renameBatch` (configSlice) cascades to **seven JSONB paths** (`syllabusBatches[]`, `syllabusBatchBranches`, `batchProgramAssignments`, `batchSyllabusProgress`, `batchChapterTimelines`, `timetables[].batchName`, `examSchedules[].batchName`) AND fires a Supabase cascade for `student_batches.batch_name` + `exams.batch` via `cascadeBatchRenameToSupabase`. The Supabase cascade is fire-and-forget (`.catch(console.error)`) тАФ caller doesn't await. Guards skip both the JSONB rename AND the Supabase cascade when oldName/newName are empty, equal, or `newName` already exists in either local list тАФ without the guard, a rejected JSONB rename would still merge rows in Supabase. Do NOT add other code paths that mutate `syllabusBatches[]` or `timetables[].batchName` directly тАФ they would bypass the Supabase cascade and re-introduce drift.
- `deleteBranch` / `deleteBatch` return `{ ok, usage }` тАФ they refuse to delete when references exist and surface the usage breakdown. UI callers must check `result.ok` and show the blocking detail; do not assume success.
- `addBatch(name, branch)` REQUIRES both parameters and a branch that exists in `branches[]`. Returns `{ ok: true, name }` or `{ ok: false, reason: 'name_required' | 'branch_required' | 'unknown_branch' | 'duplicate_name' }`. UI must check `result.ok` and surface the reason. Do not call `addSyllabusBatch(name)` directly from UI тАФ the internal action stays for migration scripts and slice-internal use only (it does not enforce the branch invariant).
- Settings page is the SOLE CRUD surface for branches, batches, and teachers. Do not add new "Manage X" buttons or modals on other pages that mutate these lists. Selection (picker dropdowns referencing existing entries) is fine; CRUD (add/rename/delete) is not. The `ManageTeachersModal.jsx` file was deleted 2026-05-20 тАФ do not re-introduce it.
- `AddTimetableModal` is select-only тАФ branch dropdown from `branches[]`, batch dropdown from `syllabusBatches[]` filtered by `syllabusBatchBranches[batch] === selectedBranch`. Do not add a free-text fallback for either field; the modal must not be able to create new branches or batches that the central lists don't know about.
- `SyllabusPage` batch tab bar is view-only. Do not re-introduce inline "+ Add batch", rename input, or the тЛп menu (Rename / Set branch / Delete). The page still owns program assignments and chapter status тАФ those are batch contents, not identity.
- `mergeStudents` discards the XLS `Batch` column тАФ existing students' `batches[]` is never modified by import; new students arrive with `batches: []`. Do not re-introduce batch-merging on import; it would undo the manual alignment sweep on every HR re-import.
- The central batch list follows `BRANCH_NDA_DURATION_(YY-YY)[_SECTION]` convention (no spaces, underscores throughout). Verify the current set against Supabase before adding to it тАФ Settings тЖТ Batches is the only UI path; direct SQL is the migration path.
- `StudentsTable`'s Aligned column / filter are gated on `centralBatches.length > 0`. Do not refactor to read from `useStore` directly тАФ the prop-based gate keeps the table testable without store mocks.
- `StudentRowEditor`'s `batchBranches` prop filters the add-batch dropdown by the row's draft branch. When `batchBranches` is `null` (legacy tests), all batches show. Keep this fallback тАФ removing it would break the existing test suite.
- `lecture_absences.lws_id` `ON DELETE CASCADE` тАФ student deletion drops their lecture-absence history. Do not change to SET NULL without first adding a `student_name` column (see `student_plans` for the precedent).
- `student_attendance.status` values are `P` / `A` / `-` / `L`. Do not add new status codes without auditing every consumer: `buildStudentStats`, `buildConsecutiveAbsent` (only counts A), `AttendanceRings` (only counts P), the import L-protection filter, and `RecentIncidents` (filters L only). New codes silently get ignored by all of these.
- `importAttendance` MUST preserve existing L rows on re-import. The L-protection block (query existing L for the imported (lws_id, date) pairs, filter them out of the upsert) is non-negotiable тАФ without it, the morning late-marking is destroyed when the LWS XLS imports at end of day. The return value's `lateProtected` count surfaces the preserved rows in the success banner.
- `setLectureAbsenteesForPeriod` uses delete-then-insert (replace-set semantics).
