# NDA Maths Tracker ŌĆö Decisions log

> **See also:** [`CLAUDE.md`](./CLAUDE.md) ŌĆö operational reference (commands, conventions); [`GUARDRAILS.md`](./GUARDRAILS.md) ŌĆö the "what not to change" invariants. This file is the long-form *why* trail for non-obvious architectural choices so they aren't re-litigated. Each row captures one decision; new entries go at the end.

| Decision | Why |
|---|---|
| `whatsappSendHistory` lives in Zustand (persisted to `faculty-data.json`), not on the exam record | It's operational state (last-sent timestamp, fail list) that must survive page refresh but is not educational data. Keeping it separate from the exam record avoids bloating exam objects and makes it easy to clear independently. |
| `failedNames` parsed client-side from script log lines (`SKIP Name ŌĆö` / `FAIL ŌåÆ Name (student`) | Simpler than changing the Python script's stdout format. Log line format is stable; parsing it in JS avoids adding a structured JSON output path that would need to stay in sync across both ends. |
| `bulkUpdateStudentContacts` does a single fetchŌåÆpatchŌåÆwrite for all edited rows | Avoids N sequential round-trips when the admin edits many students in the preview modal before sending. One read, one map, one write. |
| Subject filter is local component state, not in the store | Filters reset naturally on navigation (correct UX). No cross-page filter persistence was ever requested. Lifting to the store would add complexity with no benefit. |
| `failedNames` defaults to `null` (not `[]`) on first send | `null` means "no history" ŌĆö the preview modal uses this to decide whether to show the resend scope toggle. An empty `[]` would show the toggle but with 0 students, which is confusing. |
| `stripLatex` in `examPdf.js` outputs ASCII only (not Unicode math symbols) | jsPDF's built-in Helvetica is WinAnsi-encoded ŌĆö every Unicode symbol above U+00FF (Greek letters, set operators, arrows, `ŌēżŌēźŌēĀ`, `ŌłłŌł¬Ōł®`, `Ōł×`, `ŌäØ`, etc.) renders as garbage bytes. `├Ś` `├Ę` `┬▒` `┬Ę` (U+00D7/F7/B1/B7) are within WinAnsi and are kept. Embedding a custom Unicode font would significantly inflate the bundle; ASCII equivalents (`in`, `U`, `->`, `<=`, `alpha`, `inf`, etc.) are readable for NDA students. |
| `faculty_state` JSONB for non-exam/non-insights data; normalised tables for exams + students + insights | Exams normalised in Phase 5: 2.3 MB ŌåÆ 224 KB JSONB. Insights normalised in Phase 6 to enable history (multiple rows per student/class), per-row updates without rewriting the blob, and an FK path for future student-portal surfaces. Syllabus/timetable stay in JSONB ŌĆö no mutation gap and no `api/` endpoint reads them by ID. |
| Insights tables are insert-only ŌĆö `saveClassReport` / `saveStudentPlan` append a new row each time; the store only holds the latest | History was free with the new schema and "latest-only" would have wasted the upgrade. The Insights page reads the latest per scope via `loadInsightsFromSupabase()` to preserve the existing UI shape; full history surfaces (timeline, audit log) can be added later without schema changes. |
| `student_plans` keeps both `lws_id` (nullable FK) and `student_name` (NOT NULL) | The legacy JSONB shape was keyed by `student_name`; mapping names ŌåÆ `lws_id` at write time can fail (unresolved variants, deleted students). Keeping `student_name` always-present means a plan can be saved even when the name doesn't resolve, while `lws_id` (when populated) gives proper relational integrity. `ON DELETE SET NULL` preserves historical plans if a student is deleted. |
| `class_reports.exam_id` is nullable | Legacy reports had no exam scope; nullable column allows migrating them with `exam_id = null`. New reports written by the chat-driven flow will populate it. Schema stays permissive so we never have to backfill. |
| `migrate_insights_to_supabase.js` resolves `lws_id` from `students_db.json` (canonical name + name variants), not by querying the `students` table at runtime | The migration runs against a stable snapshot of the student data; querying Supabase per row would add hundreds of round-trips for no gain. Unresolved names are logged and inserted with `lws_id = null` so the caller can fix the variants and re-run idempotently. |
| Students use normalised tables; exams now normalised too (Phase 5) | Student mutations (batch assign, name variants, profile edits) were no-ops on Vercel because `/api/students-db` is a Vite-only dev plugin. Exam mutations had the same gap ŌĆö reads came from JSONB but writes landed only in local JSON. Both are now dual-path: Supabase table write if session active, else dev-server fetch. |
| `studentSlice.js` mutations use dual-path (Supabase if session, fetch if not) | Keeps dev workflow (local `students_db.json`) unchanged. No Vite config changes needed. The `getSession()` check is cheap ŌĆö single Supabase client call. |
| `sync:students` as a manual step for Python scripts | `send_results_whatsapp.py` reads `students_db.json`. Rather than rewriting it to query Supabase, a one-command sync keeps the Python side unchanged. Only needed before sends when student profile data changed online. |
| `saveToSupabase` is fire-and-forget (no await) | Blocking the UI on every Zustand mutation would degrade responsiveness. Supabase writes are idempotent (last-write-wins on a single row); a dropped write is recovered on next mutation or reload. |
| Supabase query chains use `async/await`, not `.catch()` | `PostgrestFilterBuilder` (Supabase JS v2) is a thenable but NOT a full Promise ŌĆö `.catch` is `undefined` on it. Calling `.catch()` throws `TypeError` silently (inside `.then()`), blocking all saves. Every Supabase query must be `await`-ed inside an `async` callback with `{ error }` destructuring. |
| `api/send-whatsapp.js` uses same URL as Vite dev endpoint | The Vite dev server intercepts `/api/send-whatsapp` before Vercel sees it. On Vercel, the serverless function handles it. No client-side URL switching needed ŌĆö `fetch('/api/send-whatsapp', ...)` works identically in both environments. |
| Vercel `BASE_URL` approach instead of hardcoded `REPO_NAME` | `import.meta.env.BASE_URL` is set by Vite at build time ŌĆö `vite build --base=/nda-tracker/` for GitHub Pages, default `/` for Vercel. Eliminates the two-file sync requirement. |
| Student login via `/api/student-login` serverless instead of static JSON files | GitHub Pages can only serve static files ŌĆö `/api/student-login` can't run there. Moving student login to a Vercel serverless function enables live Supabase queries, removes the need to regenerate `public/data/` on every deploy, and keeps student data server-side. |
| Teacher accounts use individual Supabase auth with `role='teacher'` in `user_metadata` | AES-GCM shared password was brittle (one password for all teachers) and required a static `db.json` that needed regenerating on every data change. Individual Supabase accounts give per-teacher accountability, work with the existing `onAuthStateChange` flow, and always serve fresh data via `loadFromSupabase()`. |
| `create_teacher_account.js` uses `supabase.auth.admin` (service role) not the public API | `signUp` would require email confirmation and a browser context. Admin API creates confirmed accounts instantly with custom metadata ŌĆö appropriate for provisioning internal users. Requires `SUPABASE_SERVICE_ROLE_KEY` locally; teacher accounts are created once and managed via the Supabase dashboard thereafter. |
| `migrate_exams_to_supabase.js --cleanup` prints SQL but does not execute it | The 2026-05-07 incident: cleanup ran before the tables were populated (script read local file, which had 0 exams on dev). Lesson: always seed ŌåÆ verify row count matches source ŌåÆ then run cleanup SQL manually in Supabase SQL editor after confirming the app works with normalised data. |
| `loadExamsFromSupabase` uses paginated `.range(from, from+PAGE-1)` loop | Supabase default `SELECT *` is capped at 1000 rows. With 1472+ `exam_results` rows the bare query silently cut off 472 rows, making new exams show 0 students. All queries on tables that can exceed 1000 rows must use `fetchAllRows()` pagination. |
| Exams page pagination: PAGE_SIZE = 10 | 39+ exams and growing made a single-page list hard to scan. 10 per page keeps the list short without excessive clicks. `page` state resets on filter/sort changes. |
| Attendance not stored in Zustand ŌĆö fetched on demand from Supabase | Attendance data is large (many rows per student, many students) and is read-only in the app (no client-side mutations after import). Fetching per-page avoids bloating the store and makes RLS enforcement natural. |
| `student_logins` insert is fire-and-forget (`.then(() => {})`) | Blocking the student login response on an audit write would degrade perceived performance and punish students for Supabase latency spikes. A missed login record is tolerable; a slow login is not. |
| `student_logins` table separate from `students` | Login events are operational audit data, not student profile data. Separate table keeps the `students` row small and allows efficient "last login per student" queries via the `(lws_id, logged_in_at desc)` index. |
| `StudentView` subject filter defaults to `'Maths'` | Nearly all NDA students at LWS sit Maths exams; defaulting to `'all'` caused GAT students' exams to appear alongside Maths in the default view, diluting the Maths analytics. Admin can still select `'All Subjects'`. |
| WhatsApp TRACKER_BASE hardcoded in `api/send-whatsapp.js` (not an env var) | The tracker URL is stable and public ŌĆö it's the same Vercel deployment the file lives on. Making it an env var adds a config step for no benefit. If the domain ever changes, update the constant in both `api/send-whatsapp.js` and `send_results_whatsapp.py`. |
| Attendance page has no batch filter | Consecutive absence detection is meaningful class-wide: if a student missed every class in the last N days, admin needs to know regardless of batch. A batch filter would require joining across tables or client-side filtering of a large paginated dataset. |
| `buildConsecutiveAbsent` measures streak against the **global** non-Sunday date sequence, walking backwards from the latest known date and reporting the TRUE streak length (may exceed n) | Two coupled choices. (1) Global dates: per-student "last N" would vary by student and produce incoherent comparisons; global means these students are all measured against the same reference dates. (2) True streak length: the original "last N only" version surfaced `since = earliest-of-N` even when a student had been absent for 10+ days, hiding the older absences and making "since 3 days ago" misleading. Reworked 2026-05-24 (commit `04cb061`) to walk backwards until the first P / L / missing record, so the pill says how long the absence has really been going on. |
| `/subtopic-analyse` skill reads from Supabase, not `faculty-data.json` | Tags uploaded on Vercel go to Supabase but don't sync back to the local file. Reading from the local file after a prod upload would produce stale or incomplete analysis results. |
| Subtopic renames applied via direct MCP SQL, not the migration script | For a known one-time JSONB fixup, a single SQL `UPDATE` with `jsonb_agg(CASE ...)` is atomic and requires no env vars. `migrate_subtopics_supabase.js` is retained for future runs and for team members without MCP access. |
| `studentList` stored alongside `studentProfiles` (raw vs canonical map) | `studentProfiles` is keyed by `canonical_name` ŌĆö two profiles sharing a name collapse into one entry. `FindDuplicatesTab` needs both entries to detect identical-name duplicates. Storing the raw Supabase array as `studentList` (not persisted) is cheaper than re-querying Supabase from the modal. |
| Subject selector rendered in `StudentView`'s filtered-empty-state | The default `'Maths'` filter produces zero exams for GAT-only students. Without the selector in the early-return path, those students hit a dead-end empty state with no way to switch filters. ProfileCard is also rendered for layout consistency with the other two empty states. |
| `IS_READ_ONLY` (runtime hostname check) gates dev/prod data paths in `persist.js`, `useStore.js`, `defaults.js`, `App.jsx` ŌĆö NOT `import.meta.env.DEV` | On 2026-05-20, Vercel's Vite 8.0.3 + Rolldown was confirmed to substitute `import.meta.env.DEV` with `true` in production bundles ŌĆö the opposite of correct behaviour. `loadFromDisk` was running the dev branch in prod, fetching the non-existent `/api/data` route and 404-ing. The bug survived inline refactor, version bump, and "Redeploy without cache". Switched to `IS_READ_ONLY` (a runtime hostname check that can't be miscompiled). Trade-off: dev/prod code paths can no longer be tree-shaken ŌĆö both ship in the bundle. Fix commit `7b96030`. See memory `project_vite_dev_substitution_bug`. |
| `vite.config.js` aliases `stream` to `src/stubs/empty.js` | xlsx probes `require('stream')` at module init for optional streaming support. Vite 8 externalises `stream` for the browser and its stub throws on property access (`.Readable`), turning xlsx's harmless `(stream || {}).Readable && ...` short-circuit into a startup crash (`Cannot access ".Readable" in client code`). The empty-module alias makes the optional check short-circuit cleanly. Fix commit `0127900`. |
| Student import baseline comes from `loadExistingStudents()` (Supabase if session, dev fetch otherwise) ŌĆö not from a bare `fetch('/api/students-db')` | On Vercel the bare fetch 404s ŌåÆ `existingStudents = []` ŌåÆ `mergeStudents` flagged every row "new" ŌåÆ on confirm, upsert created fresh LWS-IDs for already-existing students (silent duplicate row creation). The fix mirrors the dual-path pattern already in `studentSlice` mutations and `studentList`/`studentProfiles` loads. Bug surfaced 2026-05-20 when "21 students loaded ŌĆö 21 new" appeared after a re-import of an unchanged file. |
| `mergeStudents` matches Excel rows via tiered match (EIS ŌåÆ mobile ŌåÆ name+branch), not EIS alone | EIS-only matching meant re-registrations (new EIS issued) or EIS typos created duplicate rows for existing students. Mobile is the secondary key because it's typically stable across years. Name+branch is the tertiary because students who keep both rarely move records. Each step requires "exactly one" hit ŌĆö 2+ candidates fall through with a conflict entry so faculty can resolve manually. Step 3 requires non-empty branch to avoid false-positive matches on common single-token surnames. |
| Students page replaces (not stacks) the table with the detail view when a student is selected | Previous "Pattern X" (table + view both visible) was the original design but the user found it cluttered ŌĆö too easy to lose focus on the selected student among other names. Replaced 2026-05-20 with a single-active-view + "Back to list" button. Trade-off: filter/page state resets on return; revisit only if it becomes a friction point. |
| `deleteStudent` is hard-delete, not soft-delete | Matches the existing `mergeStudentProfiles` pattern (which hard-deletes the secondary record). Soft-delete would require `account_status='Deleted'` filtering throughout analytics, attendance, exam analytics, etc. The blast radius is guarded at the UI (`window.confirm`) and the cascade behaviour is well-defined: `student_plans` retain history via the `student_name` text column; `exam_results` (unlinked) keep the student's scores under their name string. Attendance and login audit are the only data permanently lost. |
| Role renamed `'faculty'` ŌåÆ `'admin'`; login UI unified to two tabs (Student / Admin ┬Ę Teacher) | "Faculty" and "Teacher" were both teaching staff in real life ŌĆö the previous 3-tab login was confusing about which to pick when both tabs submitted identical Supabase email+password. Renaming the privileged role to "admin" makes the distinction privilege-based (admin = read-write, teacher = read-only), and unifying the form removes the meaningless choice ŌĆö the role is decided server-side from `user_metadata.role`. The internal storage name `faculty_state` and the dev file `data/faculty-data.json` were deliberately left unchanged (no UI exposure; rename would require a Supabase migration with no benefit). The user-facing copy "LWS Pune faculty" is also retained ŌĆö it refers to the real-world teaching staff, not the app role. |
| Tiered match auto-merges only when ambiguity is zero ŌĆö ambiguous matches always fall through to insert + conflict report | Auto-picking the first ambiguous candidate would silently attach exam history / attendance / fees to the wrong student. The cost of a wrong-merge is high (data corruption that's hard to undo); cost of a "Possible conflict" preview entry is one human glance. The conservative path is correct here. |
| Tiered match keeps the "don't insert without EIS" safety net | Allowing inserts without EIS would let blank-EIS Excel rows create permanent records that lack the canonical identifier, making future re-imports impossible to match. Blank-EIS rows can still UPDATE existing students (via mobile or name+branch) ŌĆö they just can't CREATE new ones. |
| `mobile_conflict_on_eis_match` is non-blocking | The EIS match contract is "EIS wins" ŌĆö overriding it on every mobile difference would block legitimate phone-number changes. Surfacing as a conflict (rather than skipping the update) lets admin notice the rare case where two students share an EIS due to a data-entry error, without breaking the common case of a student changing their number. |
| Syllabus + timetable batch names unified by alignment, not an alias map | Two independent stores (`syllabusBatches[]` and `timetables[].batchName`) used divergent names for the same cohorts, blocking any future "filter X by student's batch" feature. Considered a `timetableBatchAliases` map and rejected ŌĆö mapping tables drift and require maintenance forever. Renaming once is a one-time cost. Direction of rename was timetable ŌåÆ syllabus (`12th Std` ŌåÆ `APJ_12th_NDA_(2026-27)` etc.) because syllabus names match the LWS/APJ-prefixed cohort convention `profile.batches[]` also uses, putting the eventual three-way join one step closer. A/B sections split on the syllabus side (`_A` / `_B` suffix) so the 1:1 mapping with timetables is preserved when both sections exist. Existing progress + timelines copied from the pre-rename cohort to both `_A` and `_B` so neither section starts blank. |
| `renameTimetableBatch` is a separate action (not part of `updateTimetable`) | The original `updateTimetable(id, patch)` does a bare patch. A bare patch is fine for changing `branch`, but a `batchName` change must cascade to `examSchedules[].batchName` (which references the timetable by name, not id). Adding the cascade to `updateTimetable` would silently broaden its semantics; a dedicated action keeps the caller's intent explicit and means the cascade can't be forgotten. |
| Settings page wraps existing slice actions rather than replacing them | Two paths considered: (a) make `branches[]` and the unified batch list authoritative with referential integrity in storage, (b) keep stores independent but add a UI layer that calls both sides' rename actions atomically. Chose (b) ŌĆö smaller blast radius, doesn't require a Supabase migration, doesn't change the wire format, can be undone by deleting one page. The trade-off is that someone calling `renameSyllabusBatch` directly (e.g. a one-off migration script) still skips the timetable side; the guardrails in CLAUDE.md document the UI-layer expectation. |
| `branches[]` ownership lives in a new `configSlice` rather than extending `timetableSlice` or `syllabusSlice` | Branches are referenced by both timetables (`timetables[].branch`, `examSchedules[].branch`) and the syllabus tracker (`syllabusBatchBranches` values). Adding `branches[]` to either of those slices would couple the wrong direction ŌĆö the slice that owns branches would need to import the other's data shape for the cascade. `configSlice` sits above both and is the natural home for cross-slice config. Future shared lists (e.g. centralised mappings) can land here without further restructuring. |
| Student `branch` aligned to central namespace via one-off SQL (2026-05-21) | After verifying the current Evalbee XLS does NOT carry a per-student branch column (so `mergeStudents` never overwrites the field on import ŌĆö [excel.js:227,253](src/lib/excel.js#L253) defaults to `''` when the column is missing, and `mergeLogic.js:142` skips empty values), 196 rows were aligned via direct SQL: `students.branch` (168 rows: 86 LWS ŌåÆ "LWS Pune", 82 APJSCH ŌåÆ "APJ") and `exams.branch` (28 rows: 18 + 10) ŌĆö `exam.branch` is set in Step 2 of the upload modal from a dropdown sourced from existing student branches, so old exams inherited the HR namespace and needed the same alignment. `students.branch = ""` (114 rows) and `exam.branch = null` (19 rows) preserved ŌĆö no signal to assign. No code changes needed; the import flow already preserves whatever is in `students.branch`. |
| `renameBranch` in `configSlice` does NOT cascade to `students.branch` or `exams.branch` | Even after the 2026-05-21 alignment, the configSlice `renameBranch` (used by Settings ŌåÆ Branches) only cascades to `timetables[].branch`, `examSchedules[].branch`, and `syllabusBatchBranches` values. A central branch rename will leave `students.branch` and `exams.branch` pointing at the old name. There is no UI surface for re-aligning them today ŌĆö it would have to be done via SQL again, or by adding a Supabase-side cascade. Acceptable for the foreseeable future because central branch renames are extremely rare (the list is `["APJ", "LWS Pune"]`). If branches ever change frequently, revisit. |
| `mergeStudents` defaultBranch option uses "fill blanks only" semantics, not "overwrite all" | The XLS doesn't carry per-student branch today, so `row.branch` is always `''` and existing students never get their branch touched by import. Adding a defaultBranch as "overwrite all" would invert that ŌĆö a wrong dropdown pick would clobber 168 correctly-aligned students. "Fill blanks only" preserves the conservative-merge rule: XLS row branch wins per-row, defaultBranch fills remaining blanks (new inserts + existing-with-empty-branch), non-empty existing branches stay untouched. Inverting requires an explicit confirm step; not built yet. |
| `studentSlice.renameBranch` / `renameBatch` removed 2026-05-21 (were shadowed dead code) | Both actions existed in `studentSlice` before the Settings page landed; the Settings work added `renameBranch` / `renameBatch` to `configSlice`. Because `useStore` spreads configSlice last, the studentSlice versions were silently shadowed ŌĆö never reachable through the store. Their only caller was the legacy `Students ŌåÆ Manage Batch/Branch ŌåÆ Rename` tab, which since 2026-05-20 has been calling the configSlice version (so a "Rename branch" click was cascading to timetables instead of renaming student records). The whole Rename tab was deleted; rename CRUD now lives only in Settings, matching the "Settings is sole CRUD surface" guardrail. |
| `branchInUseBy` / `batchInUseBy` enumerate references at call time rather than maintaining a back-index | A back-index (e.g. `branchToTimetables: { [branch]: timetableId[] }`) would need to stay in sync on every timetable/exam-schedule write ŌĆö more code, more drift risk, more for the saveToSupabase strip-list to know about. At current scale (low double-digit timetables, low triple-digit exam schedules) the linear scan is sub-millisecond and the simpler model wins. Revisit if a UI surface starts calling these in a tight loop. |
| Every batch has a branch ŌĆö `addBatch(name, branch)` requires both, no "no branch" option in the Settings UI | A branchless batch can't appear in the Timetable branch-filter or the SyllabusPage branch pills ŌĆö it's effectively homeless. The current 10-batch state was already 100% branched after the unification, so requiring branch on creation only locks in the existing invariant rather than introducing a migration. `setSyllabusBatchBranch(name, null)` is still callable (used internally by `renameSyllabusBatch` to drop a row's branch when the batch is deleted); the UI doesn't expose "clear branch" anymore. |
| Settings is the SOLE CRUD surface for branches / batches / teachers (2026-05-20) | The earlier "Settings page + existing modals as shortcuts" model meant the Manage-Teachers modal, AddTimetableModal's free-text batchName input, and SyllabusPage's tab-bar inline buttons were still drift sources. Each could introduce a name that the central lists never saw. Closing the secondary surfaces makes drift impossible to introduce through the UI ŌĆö the only paths left are scripts (one-off migrations) and direct Supabase writes, both of which are admin-deliberate. Trade-off: creating a new batch is now a two-page workflow (Settings ŌåÆ Batches ŌåÆ save ŌåÆ Timetable ŌåÆ New Timetable ŌåÆ pick from dropdown). Accepted as worth it because batch creation is rare (10 batches in 2 years) and the cost of drift is high. |
| AddTimetableModal filters the batch dropdown by selected branch | A timetable's branch and its batch's `syllabusBatchBranches[batch]` must agree ŌĆö they're the same real-world cohort. Showing all batches regardless of branch would let an admin pick branch=APJ + batch=`LWS_NDA_2Y_(25-27)_A`, which is structurally wrong. Filtering at the UI prevents the wrong combination from existing; the alternative (validate-on-save) catches it later but lets the admin form a wrong mental model in the meantime. |
| `ManageMappingsModal` (subject-teacher mappings) is NOT moved into Settings | A mapping is a `(subject, teacher, label)` triple scoped to per-timetable cells ŌĆö it's content, not a top-level config list. Moving it would either (a) make Settings the home for everything timetable-related (scope creep), or (b) split mappings from the timetable cells they belong to. Picking a teacher inside `ManageMappingsModal` is selection from the central teacher list, not editing ŌĆö that's a normal cross-reference and doesn't violate "edit only from Settings". |
| Central batch namespace cleaned to 8 names with `BRANCH_NDA_DURATION_(YY-YY)[_SECTION]` convention (2026-05-21) | The post-Settings reshape had whitespace drift inside parens (`_2Y_ (26-28)` vs `_2Y_(25-27)`), inconsistent separators (`APJ_12th NDA (2026-27)` with spaces), and a `_Std` outlier. Cleaning happened in one atomic DO block before the manual sweep so faculty wouldn't have to look at the messy names 280 times. New `APJ_9th_NDA_(26-27)` seat added in the same migration. Trade-off: the rename cascade touched 7 JSONB paths but did NOT touch `studentProfiles[].batches[]` or `student_batches.batch_name` (HR namespace, addressed separately by the sweep). |
| Batch unification ŌĆö "central as master" with manual sweep, not alias map | After listing HR-namespace batches (25 distinct, ~460 student-batch links), three approaches were considered: (a) code-side alias map (`hr_alias ŌåÆ central_batch`), (b) per-row picker at XLS import, (c) manual sweep via existing row editor. User chose (c) ŌĆö replaces three speculative phases with a one-time ~2 hour effort, and the "Aligned" pill + dropdown narrowing make drift impossible to introduce through the UI. Alias map was explicitly rejected; same reason syllabus/timetable rejected one earlier ŌĆö mapping tables drift and require maintenance forever. |
| `mergeStudents` discards XLS Batch column entirely (2026-05-21) | After the manual sweep, the next HR XLS import would re-inject HR names into `student_batches` if `mergeStudents` kept reading the column ŌĆö undoing the cleanup. Two alternatives considered: (a) "only set batches for NEW students, never touch existing", (b) "discard entirely". Chose (b) ŌĆö cleanest contract. New students arrive with `batches: []` and get assigned manually via the row editor (same flow as existing students). `parseStudentsExcel` still parses the column (callers may use it for read-only signal); only the merge step ignores it. |
| `StudentsTable` Aligned column is gated on `centralBatches.length > 0` | Lets the table stay unit-testable without store dependency (existing tests don't pass `centralBatches` and never see the column). Page-level wiring in `Students/index.jsx` reads `syllabusBatches` from store and passes it down ŌĆö production gets the column, tests get the old shape. Same pattern as the existing `isAdmin` gating: feature presence is a prop, not a runtime check. |
| Backend bulk-assign via SQL is an accepted shortcut for the manual sweep | The "manual via row editor" workflow scales to ~280 students at ~2 hours of clicking, but for large cohort moves where the central destination is unambiguous (e.g. 18 students currently in `11th & 12th Integrated - 2 Year (26-28) A` all ŌåÆ `LWS_NDA_2Y_(26-28)_A`), a direct `DELETE FROM student_batches WHERE lws_id = ANY(ids); INSERT INTO student_batches ...` via MCP SQL is faster and equally safe. Both paths end at the same outcome ŌĆö central-only batches in `student_batches`. The row editor is still the right path when faculty has to make per-student decisions (e.g. A vs B in `(25-27)`). |
| Lecture absences modelled as sparse event log (`lecture_absences`), not per-slot P/A grid | Teachers already track missed lectures in notebooks as exceptions ("Period 3 Maths: Rahul, Priya").
